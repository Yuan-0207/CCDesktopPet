## A. 微信开放平台驳回点处理（必须完成）

### A1. 官网（必填项）准备

**要求（审核口径）**

- 官网必须可公开访问（无需登录/无需加群/无需下载后才能看）
- 必须包含：应用介绍、下载方式（或应用市场链接）、基础截图/说明、联系方式
- 建议包含：隐私政策与用户协议链接（审核通过率更高）

**你需要产出**

- 1 个可访问的网址（建议：Cloudflare Pages / GitHub Pages / 腾讯云静态站 任一）
- 页面最少包含以下模块：
  - 应用名称 + 图标（与开放平台一致）
  - 应用介绍（桌宠悬浮、互动、商城、设置）
  - 4 张截图（首页、互动弹窗、商城、设置）
  - 下载（APK 直链或应用市场链接）
  - 隐私政策链接、用户协议链接
  - 联系方式（邮箱即可）

**验收**

- 用手机 4G 打开页面可正常加载
- 页面能直接看到“下载/介绍/截图/联系”

### A2. APP 流程图（信息不足导致驳回）

**要求（审核口径）**

- 需要展示完整业务流程，不是只有“登录→成功”
- 必须包含关键分支：授权同意/拒绝、异常/超时、权限开关路径
- 需明确说明：获取哪些用户信息、用途是什么、是否上传到后端

**建议流程节点（可直接画到 ProcessOn/draw.io）**

- 启动 App
- 首页展示（3D 预览 + 开关“在桌面显示桌宠”）
- 若未开悬浮窗权限：提示 → 跳转系统设置 → 返回
- 用户点击“设置”
- 用户点击“账号信息卡片”
- 进入登录页 → 点击“微信登录”
- 拉起微信授权页
  - 同意授权 → 回调 App（`WXEntryActivity`）
  - 取消授权 → 回到登录页（提示“已取消”）
- App 请求后端用 code 换取用户信息（昵称/头像 URL）
- App 保存资料（本地 `SharedPreferences`）
- 返回设置页展示昵称与头像
- 用户返回首页，继续互动/进入商城

**验收**

- 流程图中清晰标注：获取字段（昵称、头像）、用途（展示）、存储位置（本地 + 可选后端换取）、不涉及支付/分享（本期）

## C. 登录失败“可诊断化”（减少“只转圈”）

### C1. 登录状态与提示分层（用户能看懂，开发能定位）

**你已经有**

- `LoginActivity` loading mask + timeout
- `WeChatLoginEvents` 广播结果同步
- `WXEntryActivity` 后端换取用户信息并落库

**本周要补齐（建议）**

- 在 `LoginActivity` 增加一个“失败原因提示区”（不一定 Toast）
- 将失败归类成 4 类：
  - 微信未安装
  - 未收到回调（开放平台审核/体验者/签名配置问题）
  - 后端异常（Worker 返回非 200 / 解析失败）
  - 用户取消

**涉及文件**

- `app/src/main/java/com/ccpet/desktoppet/LoginActivity.kt`
- `app/src/main/java/com/ccpet/desktoppet/wxapi/WXEntryActivity.kt`
- `app/src/main/res/values/strings.xml`

**验收**

- 同一问题复现时，提示文案稳定一致
- 你能根据提示快速判断：该查开放平台还是查 Worker

## D. 真机回归用例（本周每天都跑一次）

### D1. 回归清单（10 分钟）

- 冷启动 → 首页模型显示正常
- 未开悬浮窗权限 → 开关桌宠提示合理
- 设置页 → 账号卡展示正常
- 登录页 → 点击微信登录
  - 能拉起微信（或明确提示不能）
  - 返回 App 后 loading 能停止（成功/失败/取消都必须）
- 登录成功后：设置页头像昵称刷新
- 返回首页，桌宠/互动不受影响

**验收**

- 每日记录：成功/失败次数、失败原因类别
