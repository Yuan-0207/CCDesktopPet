# 第2周执行清单（M1-1）——登录链路稳定化

> 目标：把“能拉起微信”升级成“稳定拿到昵称/头像，并且失败可定位”。

---

## A. 登录状态机与 UI（LoginActivity）

### A1. 状态机补齐

- 状态：idle / requesting_wechat / waiting_callback / exchanging / done / failed / cancelled
- 规则：
  - 进入 `onResume`：如果在等待回调，必须有“继续等待/已取消/已失败”的明确状态
  - 超时：必须停止 loading，并给出下一步操作指引（检查体验者/审核/签名）

### A2. 失败提示区（建议在页面内显示）

- 展示最近一次失败原因（可复制/可截图）
- 展示“排查建议”入口（例如：打开官网的排查说明页）

**涉及文件**

- `app/src/main/java/com/ccpet/desktoppet/LoginActivity.kt`
- `app/src/main/res/layout/activity_login.xml`
- `app/src/main/res/values/strings.xml`

**验收**

- 失败后不会只转圈
- 同类失败能输出一致提示

---

## B. 微信回调与后端交换（WXEntryActivity + Worker）

### B1. 回调必达日志（可观测）

- `WXEntryActivity` 收到回调必须能看见明确日志（tag 固定）
- `code` 是否为空的分支必须覆盖
- 取消授权分支必须覆盖并回传到 `LoginActivity`

### B2. 后端交换稳定性

- Worker 必须返回结构化错误（status + message）
- App 对非 200 的响应必须显示“后端异常”而不是“超时”

**涉及文件**

- `app/src/main/java/com/ccpet/desktoppet/wxapi/WXEntryActivity.kt`
- `app/src/main/java/com/ccpet/desktoppet/wechat/WeChatLoginEvents.kt`
- Worker 项目（Cloudflare）接口返回规范

**验收**

- 能区分：未回调 vs 后端异常 vs 用户取消

---

## C. 设置页资料回填一致性（SettingsActivity）

### C1. 回填时机与兜底

- `onResume` 必须刷新昵称/头像
- 头像加载失败要降级（默认头像），且不要卡 UI
- 对昵称为空做兜底（显示“微信用户”或“未登录用户”）

**涉及文件**

- `app/src/main/java/com/ccpet/desktoppet/SettingsActivity.kt`
- `app/src/main/java/com/ccpet/desktoppet/PetSettingsStore.kt`

**验收**

- 登录成功后，返回设置页立即看到昵称与头像
- 前后台切换不会“回到未登录显示”

---

## D. 本周真机回归用例（每天 10 分钟）

- 拉起微信授权：同意/拒绝/取消都测一遍
- 后端异常模拟（Worker 临时返回 500）：App 要明确提示“后端异常”
- 冷启动后进入设置页：资料显示一致
