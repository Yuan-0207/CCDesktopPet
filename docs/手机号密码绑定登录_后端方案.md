# 手机号绑定/登录后端方案（可落地版）

## 1. 目标与边界

- 目标：支持“本地账号 + 可选绑定手机号/密码 + 换机找回”。
- 本文聚焦后端正式方案，APP 端可先用本地模拟流程联调 UI。
- 不含短信通道采购与风控大盘实现细节（只给接入位和接口约束）。

## 2. 核心账号模型

- `user_id`：系统主键，创建本地账号时即存在。
- `device_id`：设备标识（可选），用于辅助风控，不作为唯一身份。
- `phone`：手机号（唯一，需脱敏展示）。
- `password_hash`：密码哈希（推荐 Argon2id，其次 bcrypt）。
- `password_salt`：若算法需要独立盐。
- `bind_status`：`unbound` / `bound`。
- `created_at`、`updated_at`、`last_login_at`。

## 3. 推荐表结构（最小可用）

- `users`
  - `id` (pk, string/uuid)
  - `nickname`
  - `avatar_url`
  - `created_at`
- `user_phone_credentials`
  - `user_id` (unique)
  - `phone` (unique)
  - `password_hash`
  - `password_algo`
  - `updated_at`
- `phone_verification_codes`
  - `phone`
  - `biz_type` (`bind`, `reset`, `login`)
  - `code_hash`
  - `expired_at`
  - `attempt_count`
- `auth_sessions`
  - `session_id`
  - `user_id`
  - `refresh_token_hash`
  - `expired_at`

## 4. 接口设计（V1）

### 4.1 发送验证码

- `POST /v1/auth/phone/send-code`
- 入参：`phone`, `bizType`
- 出参：`success`, `requestId`, `expireSeconds`
- 规则：60s 冷却，5 分钟有效，单手机号单日上限。

### 4.2 绑定手机号密码

- `POST /v1/auth/phone/bind`
- 入参：`userId`, `phone`, `code`, `password`
- 出参：`success`, `maskedPhone`
- 规则：
  - 验证码通过才允许绑定；
  - `phone` 已被其他用户绑定则拒绝；
  - 绑定成功后失效旧验证码。

### 4.3 手机号密码登录（找回）

- `POST /v1/auth/phone/login`
- 入参：`phone`, `password`
- 出参：`accessToken`, `refreshToken`, `userProfile`
- 规则：密码错误次数阈值后短时锁定（例如 10 分钟）。

### 4.4 绑定状态查询

- `GET /v1/auth/phone/status`
- Header：`Authorization: Bearer <token>`
- 出参：`bound`, `maskedPhone`

### 4.5 修改密码（可后续）

- `POST /v1/auth/phone/change-password`
- 入参：`oldPassword`, `newPassword`

## 5. 安全要求（必须）

- 密码不得明文存储；使用慢哈希算法（Argon2id/bcrypt）。
- 全链路 HTTPS；服务端校验时间戳与重放风险。
- 验证码在库中只存哈希，不存明文。
- 接口限流：按 IP + 设备 + 手机号三维度。
- 登录态采用短期 `accessToken` + 长期 `refreshToken`。
- 日志脱敏：手机号只打 `138****1234`。

## 6. 与 APP 的联调契约

- APP 保留本地 `localUserId`，绑定成功后由后端记录其 `userId <-> phone`。
- APP 进入“手机号登录”成功后，用服务端返回 `userId` 覆盖当前本地会话身份。
- 昵称（用户昵称）与桌宠昵称继续解耦，互不覆盖。
- 弱网/失败场景统一错误码返回，APP 仅按错误码映射文案。

## 7. 错误码建议

- `PHONE_INVALID`
- `CODE_INVALID`
- `CODE_EXPIRED`
- `PHONE_ALREADY_BOUND`
- `PASSWORD_TOO_WEAK`
- `PHONE_OR_PASSWORD_INCORRECT`
- `TOO_MANY_REQUESTS`
- `SESSION_INVALID`

## 8. 迁移上线步骤（建议）

1. 先保留本地账号能力，不中断现有用户。
2. 上线“可选绑定手机号”入口，灰度 10%。
3. 观察绑定成功率、验证码到达率、登录失败率。
4. 稳定后再放开“换机恢复”强引导。

## 9. 后端技术栈建议（低成本）

- API：Cloudflare Workers / Node.js / Ktor 均可。
- 数据库：Supabase Postgres / Neon Postgres。
- 验证码：开发期 mock，生产对接短信服务商（阿里云/腾讯云）。
- 监控：Sentry + 基础接口日志。

## 10. 当前阶段与下一步

- 当前：APP 端先做本地模拟闭环（不依赖真实后端）。
- 下一步：按本文接口补齐服务端，APP 将 mock 替换为真实请求即可。
